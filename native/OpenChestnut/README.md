# OpenChestnut

OpenChestnut is this repository's clean-room C# codec for XLSX, DOCX, and PPTX. It uses the public Microsoft Open XML SDK and the independently defined protocol in `proto/open_office/artifact/v1/office_artifact.proto`.

It is the only Office codec used by version 0.3. The JavaScript package supplies the public artifact model and wire adapter; it does not contain a fallback Office parser/writer. PDF is outside OpenChestnut.

## Projects

- `OpenChestnut.Codec` implements package validation and the XLSX/DOCX/PPTX codecs.
- `OpenChestnut.Runtime` exposes the byte-in/byte-out WebAssembly entry point.
- `OpenChestnut.Tests` covers supported creation/import/edit/export profiles, opaque preservation, and fail-closed cases.

`OpenXmlChartSpaceCodec` is the package-agnostic DrawingML ChartSpace core used
by both worksheet and slide ChartParts. The XLSX and PPTX adapters keep their
format-specific relationship, source-binding, and edit policies around that one
semantic reader/writer/patcher instead of maintaining divergent chart XML
implementations.

`OpenChestnut.Runtime` keeps its one `byte[] -> byte[]` JavaScript entry point
as a small explicit registration shim in `OpenChestnutJavaScriptInterop.cs`. This avoids
the .NET 8 `[JSExport]` generator's process-random wrapper-name hash while
retaining the same public WASM ABI. The paired clean-build gate verifies the
generated runtime file inventory and hashes rather than merely trusting an
incremental build.

Protocol version 2 removes `allow_lossy`; the removed field name and number are reserved. Opaque content can be exported only from a validated, hash-bound source package. Unsupported edits, source-evidence mismatch, unsafe OPC paths, invalid relationships/content types, and missing runtime data return structured failures.

## Supported 0.3 slices

- XLSX: cells/formulas/styles, merges/dimensions/freeze panes, passwordless worksheet editing restrictions, tables, PNG/JPEG images, charts, standard Office 2010 sparklines, canonical one-/two-variable What-If data tables, a bounded native PivotTable profile, seven data-validation rule types with bounded prompt/error/drop-down UX, six conditional-format profiles, and bounded Office 2019 threaded comments. Worksheet protection exposes all 15 operation classes as an agent-facing allow-list while the codec owns SpreadsheetML's inverted lock attributes/defaults; password/hash/extension and disabled/partial profiles remain opaque/source-bound and fail closed on replacement. Protection is not encryption or access control. Conditional formatting covers cell/formula/text/color-scale rules plus standard gradient data bars and all 17 base SpreadsheetML icon sets with typed thresholds, `showValue`, and icon reversal. x14 solid/negative-axis data bars, x14-only/custom icons, exclusive/formula thresholds, multi-area/extension validation graphs, and richer rule graphs remain opaque/source-bound. Pivot authoring owns 1 through 8 ordered row fields, zero or one column field, 1 through 32 sum/count/average/min/max value fields, optional grand totals, exact typed include/exclude filters, and optional saved cache records; multiple values use the canonical `x=-2` data-layout axis. Recognized imports are source/hash/topology-bound and read-only when editing cannot be proven safe. Compact/subtotal-bearing multi-row and multi-column graphs, non-marker scatter, noncanonical bubble profiles, richer Pivots/data tables, nested comment replies, mentions, and extended identity graphs remain opaque/source-bound.
- DOCX: styles, paragraph/run formatting, pages/sections, headers/footers, PAGE/simple fields, canonical inline five-run SEQ/REF/PAGEREF fields with an optional narrow bookmark around a SEQ cached result, one canonical one-paragraph complex TOC placeholder plus `updateFields` intent, writable `trackRevisions` intent, canonical facing-page `mirrorMargins`, per-section binding gutters plus canonical `gutterAtTop`, passwordless `documentProtection` (`none`, `readOnly`, `comments`, `trackedChanges`, or `forms`) with explicit enforcement/formatting flags, PNG/JPEG images, lists, links, classic comments plus bounded modern root/direct-reply threads with resolved/durable/UTC/person metadata, fixed-geometry tables, bounded whole-block bookmarks/internal hyperlinks, plain-text paragraph/list-item footnotes/endnotes, standalone whole-paragraph `w:ins`/`w:del` revisions, canonical inline plain-text `w:sdt`, Word 2010+ `w14:checkbox`, `w:dropDownList`, `w:comboBox`, and strict ISO/Gregorian `w:date` controls, plus one bounded bibliography `b:Sources` catalog with whole-paragraph `CITATION` fields. Irregular page-margin mode markup remains source-owned, fails closed on semantic replacement, and makes affected section geometry read-only. Protection is an editing restriction, not encryption; password verifiers, cryptographic/extension attributes, IRM, and permission exceptions remain source-owned and fail closed on replacement. The date profile owns exact `YYYY-MM-DD` visible text, UTC-midnight `w:fullDate`, `yyyy-MM-dd`, `en-US`, `date` mapping, and Gregorian calendar metadata; other date profiles remain opaque. A file-level transaction adds one exact in-paragraph replacement to original hash-bound bytes as adjacent native `w:del`/`w:ins` wrappers; it accepts a structured direct-body paragraph or bounded physical table-cell selector and requires the full paragraph/cell snapshot plus one unique literal. The literal may occupy one ordinary `w:r/w:t` or adjacent non-empty ordinary runs with identical `w:rPr`; the deletion retains every matched fragment, the insertion uses the shared formatting, and `matched_source_run_count` audits the span. Mixed formatting and broader graphs fail closed. Only `word/document.xml` changes, with target/byte/element/text/native-ID evidence returned. The table profile requires a direct body table, stable grid, non-continuation cell, and exactly one direct paragraph. Revision finalization accepts or rejects both exact-pair profiles—including same-format fragmented deletions—and the direct whole-paragraph/one-run profile, reports exact changed parts and tracking state, and rejects other story parts or complex graphs before returning output. A separately advertised `text_patchable` capability permits one source-hash-bound untracked literal replacement inside one ordinary direct `w:r/w:t` or across adjacent non-empty direct runs with byte-identical `w:rPr` in an otherwise read-only paragraph or complex table cell. The shared literal-span resolver keeps native run/text topology fixed and is also used by tracked replacement. Duplicate matches, mixed formatting, empty-run gaps, paragraph boundaries, hyperlinks, fields, controls, revisions, capability tampering, residual-topology changes, and whole-paragraph/cell replacement fail closed. Imported recognized modern threads permit text and resolved-state edits while paragraph/durable/person/UTC identity, root anchors, and topology remain source-bound. Generated modern-comment paragraph IDs and explicit inputs are constrained to the Open XML `00000001`–`7FFFFFFF` range. Imported canonical inline-field cached results, TOC placeholders, note bodies, typed content-control values, bibliography source content, and citation display text permit fixed-topology edits while field positions/instructions/bookmark identity, anchors, native identity, control type/symbols/list choices/native date profile, source order, and tags remain source-bound; imported whole-block bookmarks remain read-only. Nested/irregular modern comments, automatic field materialization, refreshed cross-paragraph TOCs, and other rich or irregular content-control/bibliography/field/bookmark/note/revision graphs remain opaque/source-bound or explicit package workflows.
- DOCX floating images: absence of a placement record remains canonical inline flow. The bounded `wp:anchor` profile is foreground, non-overlapping, layout-in-cell, absolute-offset only, and supports horizontal margin/page/column plus vertical margin/page/paragraph references, square or top-and-bottom wrap, and text distances. Recognized imports expose the marker semantically and preserve exact bytes when unchanged. - DOCX block plain-text controls: one canonical body-level `w:sdt` wraps exactly one modeled `w:p` containing one ordinary `w:r/w:t`. The public paragraph carries separate block-control metadata, so JS inspection reports block placement instead of inventing a run index. Text, tag, alias, and supported paragraph/run formatting may change after import; native ID, wrapper placement, and one-paragraph/one-run topology remain source-bound. Multi-paragraph, table/cell, nested, rich-text, locked, placeholder, repeating-section, data-bound, or extension-bearing block SDTs stay opaque and fail closed on semantic replacement.
- DOCX table-cell content controls: one canonical cell-level `w:sdt` wraps the only `w:p/w:r/w:t` in a source-free rectangular physical cell and may be plain text, Word 2010+ checkbox, standard drop-down, standard combo box, or the strict ISO/Gregorian date profile. Public handles report `placement: "tableCell"` with row/column coordinates and participate in the same typed inspect/resolve/fill/set/verify contract as inline controls. Recognized imports preserve unchanged bytes and permit only type-appropriate state plus tag/alias edits; native ID, type, cell placement, choice/symbol/date profile, and wrapper/table topology remain source-bound. Adding/removing imported controls, continuation/irregular cells, and rich, inline-within-cell, nested, multi-paragraph, data-bound, locked, placeholder, repeating-section, or extension-bearing cell SDTs stay opaque and fail closed.
- DOCX canonical list controls: source-free and recognized inline `w:dropDownList` and `w:comboBox` SDTs expose an ordered display/value table. Drop-downs accept only a declared `selectedValue`; combo boxes accept a declared choice value or bounded custom `value`. The codec derives visible text, permits only typed value/tag/alias edits after import, and source-binds choice order, values, display text, native ID, type, and run topology. Irregular list-control graphs and controls outside body runs remain opaque/source-bound.
- PPTX: direct RGB/theme solid and native style-reference slide backgrounds, text boxes/round rectangles, basic fill/line/shadow, straight/polyline connectors and arrows, source-free literal bar/line/pie/standard-area/fixed-50%-doughnut/marker-scatter/2D-bubble charts plus the bounded clustered bar+line combo, bounded embedded rectangular images with signed DrawingML `a:srcRect` crop, fixed-grid tables with canonical non-overlapping rectangular merges, recursive native `p:grpSp` trees with local `chOff/chExt` coordinates, rich text/lists/links, canonical plain-text speaker notes, native custom shows with relationship-free run hyperlinks, legacy slide annotations, bounded Office 2021 modern root/direct-reply threads with top-level element or shape-text-range anchors, source-bound Master/Layout preservation, and read-only presentation view metadata (grid spacing, snap flags, and horizontal/vertical guides). A separately advertised legacy-comment addability claim is true only when the complete imported presentation has no legacy or Office 2021 comment graph. The bounded transaction creates one canonical shared `CommentAuthorsPart` plus closed numbered `SlideCommentsPart` leaves, allocates collision-free owner relationships, preserves every SlidePart byte, and validates author/index semantics after write; an existing/mixed/connected graph, a second add, or capability tampering fails closed. The PPTX combo profile is intentionally small: literal clustered bars plus literal
lines, globally ordered series, duplicated chart-level data labels, and either
one shared primary category/value axis pair or one canonical secondary top/right
pair for every line. Bars stay primary; all lines must share one axis group.
External data, mixed line groups, secondary bars, per-series labels/point
overrides, smooth lines, trendlines, error bars, and irregular mixed plots remain
opaque-preserved or fail closed.

The other canonical PPTX chart families are equally bounded. Bar, line, pie,
standard area, and 50%-hole doughnut use literal categories and finite caches;
pie/doughnut have no axes and alone may show percentage labels. Marker-only
scatter and 2D bubble use aligned numeric X/Y caches and two value axes; bubble
sizes are positive and area-based. PPTX formula references, external or embedded
workbooks, stacked area, connected/smooth scatter, non-50% doughnut geometry,
bubble 3D/negative/custom-scale variants, and topology mutation remain
opaque/source-bound or fail closed.

Imported objects outside these modeled profiles remain opaque and unchanged. They do not become permission for lossy rewriting.

## Build and test

```sh
dotnet test native/OpenChestnut/OpenChestnut.sln --configuration Release
npm run build:open-chestnut
npm run verify:open-chestnut-build
```

The bundled consumer runtime is written to `runtime/open-chestnut`. The npm package includes that runtime, manifest, SBOM, and license notices, but excludes this C# source tree and build output. Installed consumers therefore do not need a local .NET SDK.
