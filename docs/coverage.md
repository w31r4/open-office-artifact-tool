# Coverage

This document describes the supported 0.2 boundary. It is not a promise that every legal OOXML or PDF construct is editable.

Status meanings:

- **done**: the modeled 0.2 operation is supported for creation/import, edit, export, and second import where applicable.
- **partial**: a narrower profile is editable, or imported content is preserved but some creation/editing is intentionally rejected.
- **unsupported**: no modeled operation is provided; Office export must fail rather than flatten the object if preservation cannot be proven.

## Architecture

| Surface | Status | Boundary |
| --- | --- | --- |
| XLSX/DOCX/PPTX facade path | done | All six Office facade methods lazily load the canonical OpenChestnut C# WASM adapter. There is no codec selector or JS Office fallback. |
| PDF path | done | Independent PDF implementation; never enters the OpenChestnut OOXML wire. |
| PDF provider routing | partial | Required `mupdf@1.28.0` is resolved by normal npm installation and lazily initialized on the first PDF operation. `PdfFile` plus the thin `scripts/mupdf.mjs` CLI provide the default arbitrary-file parse/inspect/render/bounded-edit route with pre-WASM byte budgets, page/object/image/render budgets, atomic distinct-path output, and no fallback. ReportLab, pdfplumber, pypdf, optional specialist PyMuPDF, Poppler, qpdf, pyHanko, veraPDF, Tesseract, and planned pikepdf/OCRmyPDF remain explicit task routes. No lifecycle installer or global mutation is used. |
| Legacy options | done | `codec`, `allowLossy`, `preferNative`, and `relativeDateAsOf` are rejected explicitly. |
| Opaque Office preservation | done | Imported unmodeled parts are content-type/hash/source bound. Unchanged content is preserved; unsupported edits and missing source snapshots fail closed. |
| Low-level OOXML inspect/patch | done | Explicit, bounded package inspection and patching remain available and are never invoked automatically as a fallback. |
| Wire protocol | done | `open_office.artifact.v1`, protocol version 2; removed `allow_lossy` field name and number are reserved. |
| JavaScript source layering | partial | `src/index.mjs` is now a 49-line composition root: it preserves the 41-symbol public API, cross-format kind inference, verify/render/visual-QA/help wrappers, exact model binding re-exports, and five font-metric exports while the complete PDF, Document, Presentation, and Spreadsheet model/layout/render/file-facade domains live in dependency-leaf modules. Visual QA receives artifact-kind inference through one explicit factory boundary, so it has no model back-edge or circular import. Both OpenChestnut Office adapters import their Presentation or Spreadsheet leaf directly instead of creating root-entry back-edges. The shared OOXML module owns JSZip package loading, safety budgets, OPC paths/content types/relationships, recipes/source references, validation, and transactional regeneration behind four internal exports. Root PDF, Document, all eight Presentation, and all four Spreadsheet bindings are strict-identical leaf re-exports, and the cross-format ID allocator remains a single shared module. The root is 49 of 28,642 JavaScript source lines (0.2%); future decomposition concerns large but cohesive domain leaves, not the public composition root. |
| Font inventory and design metrics | done | `document.fontFamilies`, `workbook.fontFamilies`, and `presentation.fontFamilies` return fresh, sorted, case-insensitively deduplicated inventories of the model's relevant theme/default/explicit fonts. The public metric registry supports replacement, last-in scoped registration with idempotent disposal, primary-family/style/nearest-weight resolution, clearing, and deterministic Skia subpixel-baseline compensation; invalid metric records are ignored and unsupported family fallback is never silent. |
| Agent black-box PromptBench | partial | The repository-only v1 suite defines 26 realistic prompts (19 PDF, 7 Office), isolated candidate/reference Skill trials against the same packed npm candidate, dirty-worktree plus tarball provenance, immutable input fingerprints including directory trees, exact deliverables, fail-closed outcomes, and generic hard gates. All seven ready PDF cases now add independent structural/semantic evidence, Poppler/Pillow all-page comparison where applicable, residue/revision/path-safety or accessibility-claim gates, audit provenance, and agent trace-discipline grading. The attachment fixed matrix passes candidate `3/3` at 100/100 and reference `2/3`; AcroForm similarly passes `3/3 + 2/3`, while active-content, greenfield accessibility, and merge/reorder/selective-watermark each pass `3/3 + 3/3`. The 19 corpus/PKI cases await pinned licensed assets. |
